WARD VISITOR SYSTEM Blueprint v9
Blueprint v9 — Core System
Ward Visitor Management System
Refines v8 following a conversation with a research-level nurse. The dashboard moves from an open, any-staff-member model to a single coordinator role (receptionist, or a covering coordinator), reflecting standard NHS PC auto-logout and accountability practice. Adds silent bay-movement handling, an "I don't know the bay number" option, and a returning-coordinator summary so missed time at the screen is never a missed visitor. EOL handling moved to the roadmap pending clinical input. Electronic door release removed from scope entirely.
Project repository and contact information see README.md: GitHub
Design principle
The nurse at the bay is always the decision-maker. This system's only job is to make sure that decision gets triggered reliably — and that the visitor always knows their arrival has been registered, even if staff are occupied. It must add no meaningful burden to staff or visitors.

Step 3 is the core change from the current system. The visitor receives confirmation before any staff action has occurred — and never learns of a bay change from the kiosk. Step 4's audible alert is what makes a single-coordinator model workable despite standard NHS screen auto-logout.
What the system is

① Visitor Kiosk

One per entrance. Tablet in full-screen browser mode. Two inputs: first name and bay number — or "I don't know the bay number". Immediate confirmation on submission, with no bay number ever echoed back. Displays restricted period notices and staff messages. Switches to visiting-ended screen at nightly cut-off.

② Coordinator Dashboard

Browser at the reception or coordinator desk. Live queue of waiting visitors with an audible alert on arrival, plus a returning-coordinator summary on login so a brief absence from the screen — including standard NHS auto-logout — does not mean a missed visitor. Handles admission, messaging, bay-movement redirects, restricted periods, and ward configuration — one screen, one role, no separate admin panel.

Who uses the dashboard
Receptionist, or a covering coordinator when reception is unstaffed — typically someone at coordinator level who sits in the same area and can pick up the function during leave, sickness, or breaks. This replaces the earlier "any staff member, any ward PC" model. Two reasons drove the change: ward PCs auto-log out as standard NHS security practice, so a nurse glancing at a locked screen would receive no information; and a single named role brings clearer accountability than an open, unassigned queue.
What the system is not

Current failures addressed

✓
Visitor always gets immediate acknowledgement — never left without information
✓
Audible alert means a screen left briefly (or auto-logged out) does not mean a missed visitor
✓
Escalating alerts — requests that age are flagged prominently
✓
Full timestamped log — who arrived, waited, was admitted, by whom
✓
Nightly automatic reset — clean state each morning, no staff action needed
✓
A patient's bay move is never disclosed to a visitor by the system — handled silently, told to the family in person as normal

Pilot architecture — preferred approach
Preferred pilot architecture: self-contained ward appliance. Run the entire system on a small local server (Raspberry Pi or equivalent mini PC) with its own private WiFi network — a travel router or the Pi's own access point. The kiosk tablet and the coordinator desk PC connect to this private network only.

This single decision dissolves the main risks flagged in earlier versions:

✓ No NHS WiFi dependency — no captive portal problem, no trust IT negotiation for a VLAN
✓ No data leaves the ward — no cloud hosting, no HTTPS/domain dependency for the pilot, no data residency question
✓ IG conversation is simpler — a sealed box on the ward holding first names for a few hours is a much shorter conversation than cloud-hosted personal data
✓ Total hardware cost approx £150–250 — Pi/mini PC, travel router, tablet, mounts

If the Pi approach proves impractical, HTTPS over NHS WiFi remains viable but requires trust IT involvement. See the Technical tab for both options.

Kiosk design principles

✓
Two inputs, or one plus a flag — first name and bay number, or first name alone via "I don't know the bay number". Nothing else asked.
✓
Immediate confirmation — screen updates before any coordinator action.
✓
Never echoes a bay number back — this is the mechanism that keeps bay-movement redirects invisible to the visitor.
✓
First name only stored — enough for staff to address the visitor and personalise messages. No surnames.
✓
Language selection — visitor-initiated, persists for the session.

Auto-resets after inactivity — ready for the next visitor without staff intervention.

System Logic
Rules & Logic
Simple by design. The fewer rules the system enforces, the less it can get in the way.
Core rules
①
No visitor is ever left without acknowledgement. The kiosk confirms receipt immediately on submission, before any coordinator action has occurred.
②
One named role handles the queue. Receptionist, or a covering coordinator when reception is unstaffed (leave, sickness, breaks) — usually someone at coordinator level in the same area. Not open to any staff member, since ward PCs auto-log out as standard NHS practice and an unattended, unlocked dashboard would not be appropriate.
③
Two-part mechanism covers absence from the screen. An audible alert plays when a new visitor registers, for when the coordinator is within earshot. But sound alone does nothing if they are genuinely away — at the far entrance, with a patient, on a break — so on every return to the screen, including after standard NHS auto-logout and re-login, a "while you were away" summary shows what was missed: count of new arrivals and the longest current wait. Together these mean a screen left unattended for any length of time is never the same as a missed visitor.
④
The nurse at the bay makes all admission decisions. The system surfaces the request to the coordinator, who relays it to the bay. It does not enforce or automate the decision.
⑤
The kiosk never confirms or echoes back a bay number to the visitor. This is deliberate — it is what allows a bay-movement redirect to remain invisible to the visitor at the point of arrival.
⑥
Bay-movement redirects are silent and temporary. If a patient has moved bay, the coordinator sets a redirect (old bay → new bay). A visitor entering the old number is reassigned automatically; the system tells them only that the ward has been notified. The redirect expires automatically after 24 hours. The family is informed of the move in person, in the normal way — the system never discloses it.
⑦
"I don't know the bay number" triggers no lookup. The system does not attempt to match a visitor to a patient. It simply flags the visitor to the coordinator as needing in-person help.
⑧
Restricted periods pause admissions but not registration. Visitors can still register during a ward round. They are queued and admitted when the period ends.
⑩
Nightly reset is automatic. At the configured cut-off time, all active sessions are closed and bay counts reset. Kiosk switches to visiting-ended screen. Visit log is retained. Bay-movement redirects are unaffected by the reset and continue running on their own 24-hour expiry.
⑪
Visiting resumes automatically at the configured start time. Kiosk switches back to the arrival screen. No staff action needed.

Privacy note
First name only. The system stores visitor first names and bay numbers only. No surnames, no patient names, no clinical information. Bay number is the sole link to patient location. Storing first names only rather than full names meaningfully reduces IG weight — a system holding first names for hours on an air-gapped ward device is low-risk personal data processing. A DPIA may still be required — confirm with the trust's IG team.
Architecture
Technical Architecture
How the system is built, hosted, and deployed. Designed to be as simple as possible while remaining secure and maintainable by trust IT.

No heavy frameworks needed. Vanilla JS on the frontend. Node.js + Express on the backend. Socket.io for real-time updates. This stack is widely understood, easy to hand over to trust IT, and has no licensing cost.
Hosting — pilot vs production

Pilot hosting — preferred: ward appliance

①
Raspberry Pi or mini PC on the ward
Runs Node.js locally. Serves the app to kiosk tablet and ward PC over its own private WiFi network (travel router or Pi access point). Nothing touches NHS WiFi or the internet. All data stays physically inside the locked ward.
Preferred
②
Private SSID — no captive portal
The Pi or travel router creates its own WiFi network. Kiosk tablets connect automatically with no sign-in page. No trust IT involvement needed for the network itself.
Clean solution
③
Approx hardware cost: £150–250
Pi 4 or mini PC (~£60–100), travel router (~£25–40), tablet (~£80–120), mounts and cables (~£20). One-off cost. No ongoing hosting fees.
Low cost
④
Fallback: VPS over NHS WiFi
If the Pi approach is impractical, a small UK-based VPS (~£5–10/month) with HTTPS via Let's Encrypt works over NHS WiFi. Requires captive portal testing and raises the IG conversation about data leaving the building.
Fallback only

Production hosting — longer term

①
Trust-hosted infrastructure
The correct long-term answer. Trust IT hosts the server on internal infrastructure or an NHS-approved cloud platform. Longer to arrange but keeps everything within trust governance.
For production
②
NHS-accredited cloud (e.g. N3 / HSCN)
NHS has approved cloud platforms. Trust IT will know which apply. Required for any system handling patient-adjacent data at scale.
For production
③
GitHub / personal web hosting
GitHub Pages is static-only — cannot run a backend. Personal hosting is not appropriate for patient-adjacent data even in a pilot.
Not suitable
Network & WiFi security
⚠
Open NHS hospital WiFi — public network
Anyone on the same network can see unencrypted traffic. HTTPS mitigates this for data in transit. However the staff dashboard on a public network is a concern — visitor names and bay numbers are visible on screen and in transit.
Needs resolution
①
Kiosk tablets on open WiFi
Acceptable for the pilot. Tablets submit name and bay number and receive confirmation. HTTPS encrypts this. Risk is low.
Acceptable for pilot
②
Staff dashboard on open WiFi
Higher risk. Ideally accessed from trust staff WiFi or wired network. Worth raising with trust IT before pilot begins.
Raise with IT
③
Captive portal on NHS WiFi
Open NHS WiFi often requires a sign-in page. Tablets in kiosk mode may not handle this automatically — they may get stuck on the sign-in page rather than the kiosk app. Needs testing before pilot. A dedicated ward VLAN or network would solve this cleanly.
Test before pilot
④
HTTPS — mandatory
All traffic between browser and server must be encrypted. Free via Let's Encrypt. Non-negotiable regardless of network used.
Mandatory

Future Phases
Roadmap
Features deliberately excluded from v1 to keep the core system simple and deliverable. Each is well-understood from the design process and can be added once the core system is stable and trusted by staff.
Phase 2 — Visitor management

Visitor Sign-Out Phase 2

Visitor taps "I am leaving" on the kiosk on departure. Records departure time. Staff can manually clear a visitor from the dashboard.

!
Known problem: visitors stepping out temporarily who forget to sign out will be locked out on return, requiring staff intervention. This must be solved before sign-out can gate bay capacity.

Real-Time Bay Capacity Tracking Phase 2

System enforces maximum 2 visitors per bay in real time. Depends on reliable sign-out. Bay grid on dashboard shows live occupancy rather than "admitted today".

!
Cannot be reliably implemented without sign-out. Premature enforcement creates more staff interruptions than it saves.

End of Life Bay Handling Phase 2

Some form of priority flagging for EOL visitors arriving at the door. Removed from v9 entirely rather than left as a simple flag, because a brief conversation with a research-level nurse suggested current practice is likely quite different from a routine "flag and prioritise" model and may not be something a queue-based system should touch without proper input.

!
Needs a real conversation with clinical staff with current HDU/ITU experience before any design work resumes — not assumptions carried over from a different design context.

Lock-Screen Visitor Count Phase 2

A waiting-visitor count visible on the NHS PC's own lock/login screen, before any credentials are entered — so a coordinator glancing at a locked screen from a distance still gets a signal.

!
Depends on whether trust IT can customise the lock screen of a standard NHS PC, which is unknown and may not be possible at all. The returning-coordinator summary in v9 covers the same need without requiring this.

Smart Re-Entry on Login Phase 2

Dashboard automatically scrolls to or highlights the longest-waiting or highest-priority visitor the moment a coordinator logs back in, rather than just showing a summary banner.

!
A refinement of the v9 returning-coordinator banner once real usage shows whether a passive summary is enough or whether something more directive is needed.
Phase 3 — Access control

Approved Visitor Lists Phase 3

Per-patient list of approved visitors, built at the bedside by patient or next of kin with nursing assistance. One designated next of kin. Unlisted visitors held pending verification. Ward sister / nurse manager has override authority. List moves with the patient. Deleted on discharge.

!
Requires information governance and legal review before build. Trust must clarify the distinction between next of kin and Lasting Power of Attorney, and define a patient consent process.
Phase 4 — Extended reach

Multi-Ward Deployment Phase 4

The system is already designed to be configurable per ward. Deployment to additional locked wards requires only ward-specific configuration — no code changes. A trust-wide view across all wards could be added for matron-level oversight.

Reporting & Analytics Phase 4

The visit log accumulates rich data over time: peak visiting hours, average wait times, frequency of long waits, busiest days. This could inform staffing decisions and ward management. Requires only a reporting view on top of the existing log.
